# Design: v2.2 Cleanup — retract unshipped scaffolding, finish v2.0 redesign

## Overview

The v2.0 redesign (Phases 26-46) is structurally complete but left behind two classes of debt:

1. **Unshipped scaffolding** — `policy-v2/`, `schemas/config-v2.ts`, and the `migrate` command that writes a shape `loadConfig()` cannot read. Docs (AGENTS.md, SECURITY.md, parts of SPEC) describe this as the new convention, but it never wired through.
2. **Forgotten cleanup** — removed CLI commands (`verify`, `link`, `unlink`, `enable`, `disable`) still wired in code or completions; dead modules (`httpAdapter`, `searchSkillsRegistry`); legacy fallback paths (`info.ts`); CI scripts referenced from `package.json` but not committed; foundation docs still describing pre-redesign behavior.

This design lands as **v2.2.0** — a minor bump from v2.1.1, not a major. **No user-facing breaking changes**: the legacy `[[security.overrides]]` + `preset` config shape that users actually have keeps working unchanged. The fix is to retract the unshipped V2 scaffolding (Direction B), delete dead code, complete the in-flight cleanups, and align internal foundation docs to the actually-shipped surface.

### Direction decision

**Chosen: Direction B (retract V2 scaffolding, document legacy as canonical).**

Direction A (wire `ConfigV2Schema` + `policy-v2/` through `loadConfig`/`composePolicy`, deprecate legacy `overrides`/`preset`) was rejected because:
- It's a breaking change to user configs (would require a re-migration window).
- The user explicitly framed this as "a minor update, not v3.0".
- It's significantly more work (rewrite `composePolicy`, `config security` CLI, all consumers, migrate output, plus a new round of doc alignment).
- The legacy system already works; the V2 schema's only documented improvement (`trust = [...]` glob array) is achievable with a future minor that wires `policy-v2/trust-glob.ts` if/when it's prioritized.

If the team later wants to ship the V2 simplification, Direction A is a separate v3.0 design — out of scope here.

### Unit 1.2: Rewire `migrate/config-v1.ts` to write `ConfigSchema` shape

**File**: `packages/core/src/migrate/config-v1.ts`

The migrate command currently translates v1 (per-mode `[security.human]`/`[security.agent]` + `[agent-mode]`) → `ConfigV2Schema` (flat with `trust = [...]`). After this unit, it translates v1 → `ConfigSchema` (flat with `[[security.overrides]]` + `preset`), which is what `loadConfig()` actually parses.

**Translation rules (new)**:
- `[security.human].scan` + `[security.agent].scan` → `security.scan` (take stricter: `semantic` > `static` > `off`)
- `[security.human].on_warn` + `[security.agent].on_warn` → `security.on_warn` (take stricter: `fail` > `prompt` > `allow`)
- `[security.human].require_scan` + `[security.agent].require_scan` → `security.require_scan` (take true if either)
- `[security.agent].agent_cli` (or shared) → `security.agent_cli`
- `[[security.overrides]]` (v1) → `[[security.overrides]]` (kept verbatim — same schema; Phase 40 collapsed per-mode but kept overrides). Drop the warning that today says "translated to trust=[...]".
- `[agent-mode]` block → dropped with warning ("agent-mode config block was retired in v2.0; pass --yes/--json for non-interactive use").


## Risks

### R1: `info.ts` fallback removal silently degrades for unmigrated users
The legacy fallback was the only way `skilltap info <name>` worked for v0.x users with `installed.json` and no `state.json`. After Unit 3.1, those users see "not installed" until they run `skilltap migrate`. **Mitigation**: install/update commands already show a soft startup warning for v0.x users. Document the change in the v2.2.0 changelog under "behavior changes". If support burden is high, restore the fallback as a `--legacy` flag in v2.2.1.

### R2: Migrating `[[security.overrides]]` verbatim may not produce the user's intent
The original v0.x per-mode model had `[[security.human.overrides]]` and `[[security.agent.overrides]]`. v2.0 collapsed to flat `[[security.overrides]]`. If a user had different overrides per mode, Unit 1.2's "verbatim" translation needs to merge them. **Mitigation**: explicitly handle the case in Unit 1.2 — when both per-mode override arrays exist, concatenate (preferring the stricter mode if a `match` collision happens) and emit a warning recording the merge.

### R3: Foundation doc rewrites (Unit 4.2 SPEC excision especially) are large and may introduce regressions in cross-references
SPEC.md is the most-cited internal doc. Cutting 2900 lines may break inline anchors used by other docs. **Mitigation**: after Unit 4.2 lands, grep all docs for `SPEC.md#` anchor references and fix or update them in the same commit. Spawn a Sonnet agent for the actual rewrite to keep the orchestrator focused on cross-doc consistency.

### R4: Phase-marker stripping (Unit 3.8) may delete intent-bearing comments
Some phase markers explain non-obvious behavior (e.g., "smart-default per Phase 31c-c-1 — see X"). **Mitigation**: keep up to 5 markers if they add real context. Each preserved marker gets a one-line follow-up explaining the relevance.

### R5: User scripts using `skilltap verify` will break
Unit 2.1 deletes the command. The error-with-hint mechanism mitigates interactive use, but CI scripts will hard-fail with non-zero exit. **Mitigation**: document in v2.2.0 changelog under "Breaking" — recommend `doctor skill` as the replacement. Keep the error message machine-greppable.
